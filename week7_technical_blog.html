SQL & NoSQL

Date: June 6th, 2014

Compare SQL databases with NoSQL databases. What are the pros and cons of either?

SQL Databases:

SQL, in which I pronounce it see-quel and is short for structured query language, is a programming language specifically for managing data in "relational database management system" (RDBMS). The basics of this language is to be able to interact with a database, like making queries that select specific data we want. For example, when I was a kid, the library had computers that prompted the user in a command line fashion to make a query about what they were looking for. This data management system allowed the user to input in fields of the author, book title, ISBN, or keyword queries. However, these fields are preset, and can not make very detailed searches, like selecting multiple queries to a book, like publication year and publisher at the same time, since we only had preset fields. Remember, this was on IBM machines in the early 90's running MSDOS. To do this, we can run a SQL query to select this information.

A relational database is pairing tables to other tables. These relationships can be one-to-one, one-to-many, and many-to-many, as we learned in one of our exercises. For the purposes of this article, we are comparing relational and non-relational databases.

Pros:

  • For Oracle, SQL does scale (this can be a con in previous versions)
  • Can be paired with node.js

Cons:

  • Need to servers during growth as data and processing expands
  • Does not process unstructured data
  • Non-dynamic, fixed schemas (but within a certain context)

NoSQL Databases:

NoSQL, short for Not Only SQL, is a type of database that is not relational, is distributed, open-source, and horizontally scalable. This structure came up as an alternative to the relational database model because of the amount of data that may need to be collected by mobile users, like location data, purchase data, wallet data, etc. Unstructured data allows these transactions to be faster without having to structure everything. Also, these databases are more dynamic and do not need to conform to a strict and defined schema.

Pros:

  • Scalability
  • Processes unstructured data
  • Cloud-based (or cheaper) storage
  • Can be paired with node.js

Cons:

  • No guarentees of ACID (atonomicity, consistency, isolation, durability)
  • Does not use tables
  • No SQL support

My Thoughts:

Honestly, which one I prefer just depends on the context of the thing I'm trying to build. Because we are learning Ruby and Rails primarily in DBC, and we want things to be more dynamic, I want to mess around with NoSQL a lot more. It seems in terms of context, we will be using the cloud for storage and servers that are not in-house.
