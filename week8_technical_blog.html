Free-For-All Blog Topic: Big O Complexity and Notation

Date: June 11th, 2014

This week's technical blog is a free-for-all. I looked at the list of topics, and being too lazy to choose my own, I've decided to talk about Big I Complexity and Notation. The decision came about because it was something I had never heard about and something I haven't written about before (I've written a blog post on Web Security a few weeks ago and I've written a bit about regular expressions).

Complexity

Even with all of our processing power, memory and network advances over the past few decades, we don't want to waste resources from inefficiency. In my simulations class in college, the professor lectured about inefficiencies in algorithms. Back in the day when memory was really expensive (relative to today's storage capacities), you wanted code that could give the desired output in the lowest time cost. More processing time meant fewer simulations. We were given the principles behind refactoring, but we never utilized them.

We need to differentiate performance and complexity. Performance is characterized by how much time, memory, disk space, and other processes are used when the program is running. Complexity involves scaling. How do resource requirements change when the algorithm grows? What happens when the problem being solved becomes larger? Complexity affects performance, but not vice versa.

The time required by a method is proportional to the number of basic operations it must perform (e.g. one test, one read, one write, one arithmetic operation). When the method performs the same tasks every time they are called, we call this constant time.

When methods are performing different number of operations, the performance changes. Two factors affect the number of operations being performed: problem size and input size. Looking at our algorithms, we need to ask how does the number of operations relate to the problem size? For simple tasks, like the flatten method in Ruby, the same operations will be performed everytime. For more complex methods that include blocks, procs, or lambdas, this could change the number of operations being performed. In the linear-time method, if the problem size doubles, the number of operations also double. In the quadratic-time method, if the problem size doubles, the number of operations triples, or quadrupals, or is something larger than doubles. When accessing complexity, we want to look at the worst case, or methods with high complexity.

Big-O Notation

Complexity is expressed through the Big-O Notation. I took the following list from a class at the University of Wisconsin-Madison. Link found in resources.

  • a constant-time method is "order 1": O(1)
  • a linear-time method is "order N": O(N)
  • a quadratic-time method is "order N squared": O(N2)

For the big-O notation, we're looking at the worst-case scenario. As mentioned above on how this works, we look at the method that will take the most time and that will be our order for the algorithm. There's a fourth notation, which is O(log N) which explains a time method scaled logorithmically. When the input size grows, the number of operations may be a lot less. One example is binary searches, which follows this pattern: "an input data set containing 10 items takes one second to complete, a data set containing 100 items takes two seconds, and a data set containing 1000 items will take three seconds." (Rob Bell)
